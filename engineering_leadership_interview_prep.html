Coding Standards & Engineering Best Practices
What engineering leaders are expected to know and advocate for
🧹
Clean Code
Functions do one thing. Meaningful names. No magic numbers. Small functions. Comments explain WHY, not WHAT. Code should read like prose.
🔷
SOLID
Single Responsibility · Open/Closed · Liskov Substitution · Interface Segregation · Dependency Inversion. Know examples for each in JS and Go.
🔄
DRY / KISS / YAGNI
Don't Repeat Yourself · Keep It Simple, Stupid · You Ain't Gonna Need It. The anti-over-engineering trio. Know when to break DRY intentionally.
🧱
Modularity
High cohesion, low coupling. Separation of concerns. Layered architecture (controller → service → repository). Dependency injection.
⚠️
Error Handling
Fail fast, fail clearly. Never swallow errors. Return errors, don't panic. Wrap errors with context. Distinguish operational vs programmer errors.
🧪
Testing
Unit + integration + e2e pyramid. TDD mindset. Tests as documentation. Avoid test coupling to implementation. Mock external deps.
🔍
Code Reviews
Review for correctness, readability, security, performance, test coverage. Be kind, be specific. Approve what's good, not perfect. Keep PRs small.
🔁
Refactoring
Leave code better than you found it (Boy Scout Rule). Refactor in small steps. Use IDE tools. Don't refactor and add features simultaneously.
🌐
API Design
REST: resource-based, proper HTTP verbs/status codes. Versioning strategy. Idempotency. Pagination. Clear error contracts. OpenAPI/Swagger docs.
🔐
Secure Coding
Input validation. Parameterized queries. Least privilege. Secrets in env vars. Rate limiting. OWASP Top 10 awareness. Dependency auditing.
📊
Observability
Structured logging (JSON). Distributed tracing (trace IDs). Metrics (latency p99, error rate, saturation). Dashboards. Alerts. On-call runbooks.
🌿
Git & PR Hygiene
Atomic commits. Conventional commit messages. Short-lived branches. Small PRs (<400 LOC). Trunk-based development. Feature flags over long branches.
JavaScript Best Practices
Prefer const/let over var. Use strict equality (===). Avoid implicit coercion. Use optional chaining (?.) and nullish coalescing (??).

Async patterns: Prefer async/await over raw Promise chains. Always handle Promise rejections. Use Promise.all for parallel ops. Never use async in forEach — use for...of or Promise.all with map.

Functional style: Prefer map/filter/reduce over imperative loops for readability. But don't sacrifice clarity for cleverness.

Common pitfalls: Closure in loops (use let or IIFE), this binding in callbacks (use arrow functions or .bind()), floating point arithmetic, event loop blocking with synchronous heavy computation, memory leaks from event listeners not removed.

Module system: ESM (import/export) preferred. Know CommonJS (require/module.exports) for Node.js legacy. Avoid circular imports.

Performance: Debounce/throttle UI events. Avoid DOM manipulation in loops. Use WeakMap/WeakSet for memory-friendly caching. Profile with Chrome DevTools.

Interview signals: Knowing the event loop and microtask queue. Explaining closure, prototype chain, and class inheritance. Understanding how V8 JIT compiles hot paths.
Idiomatic Go Best Practices
Error handling: Return errors as values. Wrap with fmt.Errorf("context: %w", err). Check errors immediately. Never ignore errors with _. Use errors.Is and errors.As for type checking.

Interfaces: Keep interfaces small (1–3 methods). Define interfaces at the consumer side, not the producer. Use io.Reader, io.Writer, fmt.Stringer as models. Composition over inheritance.

Concurrency: Don't communicate by sharing memory — share memory by communicating. Use channels for coordination, mutexes for state. Use context.Context for cancellation and deadlines. Always close channels from the sender side.

Goroutines: Never leak goroutines. Use sync.WaitGroup for fan-out. Use errgroup for goroutines that return errors. Use select with done channel for cleanup.

Package structure: Group by domain, not by type. Avoid package util or common. Use internal/ for private packages. Keep main package thin — it wires dependencies.

Testing: Use table-driven tests. Use testify for assertions or standard testing package. Use interfaces to mock dependencies. Benchmark with testing.B. Use -race flag to detect data races.

Performance: Pre-allocate slices with make([]T, 0, n). Avoid string concatenation in loops — use strings.Builder. Use sync.Pool for high-frequency allocations. Profile with pprof.
📏
Engineering Metrics (DORA)
Deployment Frequency · Lead Time for Changes · Change Failure Rate · Mean Time to Recovery. Know your team's baseline and how to improve each.
🚨
Incident Management
Severity tiers (P0–P3). On-call rotation. Incident commander role. Blameless post-mortems. 5 Whys. Action items with owners and dates.
🔧
CI/CD Pipeline
Fast builds (<10 min). Automated tests gate merges. Staged deployments (canary/blue-green). Rollback mechanisms. Feature flags for safe releases.
🏗️
Infrastructure as Code
Terraform / Pulumi. Environments as code. Immutable infrastructure. GitOps. No manual console changes in production. Drift detection.
🛡️
Security Posture
Threat modeling. Dependency scanning (Snyk, Dependabot). Secret scanning in CI. RBAC. Audit logs. SOC 2 / ISO 27001 awareness for leadership.
📚
Documentation
ADRs (Architecture Decision Records) for major decisions. README-driven development. Runbooks for ops. API documentation. Onboarding docs.
Pair Programming Preparation
How to think, communicate, and code like a senior engineer in a live session
🗣️
Narrate Your Thinking
Never code in silence. Say what you're considering: "I'm thinking a HashMap here because lookups would be O(1)…" Talk through trade-offs before typing.
❓
Clarify First
Always spend 2–3 minutes clarifying: input types, edge cases, constraints, expected output format. Ask: "Can the input be null? What's the max size?" Shows engineering maturity.
📐
Structure Before Code
Write a plan in comments first: // Step 1: Validate input // Step 2: Build the map // Step 3: Traverse. This anchors both you and the interviewer.
🔄
Iterative Approach
Start with a brute force. Say "I'll start naive then optimize." Then improve. Shows you can ship working code and then refine — exactly what seniors do.
🤝
Receive Feedback Gracefully
If the interviewer hints, say "That's a good point — I think you're suggesting X, let me try that." Never be defensive. They're testing collaboration, not just code.
🔧
Recovery Protocol
When stuck: pause, re-read the problem, consider edge cases, try a smaller example. Say "Let me think about a simpler version first." Silence for 30 sec is fine.
✅
Test Your Code
After writing, trace through with an example input. Test edge cases (empty array, single element, max value). Say "Let me verify this with the example given."
⚡
Production Mindset
Name variables meaningfully. Handle errors. Add comments. Ask: "Should I add logging here?" Shows you think about code that runs in production, not just passes tests.
🎙️ Verbal Templates for Pair Programming
Opening a problem
"Before I start coding, let me make sure I understand the problem correctly... [restate]. Is that right? And are there constraints on the input size? Should I handle null inputs?"
Starting to code
"I'm going to start with a brute force approach first — O(n²) — just to have something working, then I'll optimize. Does that sound okay?"
When stuck
"I'm not immediately seeing the optimal path here. Let me think about what structure would help... A sliding window maybe? Or could this be a two-pointer approach?"
After finishing
"Let me trace through with the example: input is [2,7,11,15], target 9... we'd find 2, look for 7 in the map, it's there... returns [0,1]. Looks correct. Time complexity is O(n), space O(n). Any edge cases you want me to handle?"
Engineering Leadership Interview Prep
STAR stories · Common questions · Answer frameworks
⭐ STAR Framework for Leadership Stories
Use STAR for every behavioral question. Aim for 90-second answers. Quantify impact always.
S
Situation
Set the context in 1–2 sentences. Team size, company stage, what was the environment?
T
Task
What was YOUR specific responsibility? Not the team's — yours. What were you accountable for?
A
Action
What did YOU specifically do? 3–5 concrete actions. Use "I" not "we". Show your decision-making.
R
Result
Quantify the outcome. Reduced latency by X%. Delivered Y on time. Promoted Z engineers. Business impact?
📝 12 Must-Prepare STAR Stories
  • A technical decision you made that had significant impact (positive or negative)
  • A time you managed an underperformer — what you did and how it ended
  • A major incident you led — how you handled pressure and the post-mortem
  • A time you pushed back on a product or business decision technically
  • A cross-functional conflict you resolved — between eng and product, or eng and eng
  • A delivery that was at risk — how you recovered it
  • A time you influenced a team or org without authority
  • A time you set or changed engineering culture/practices
  • Your biggest technical mistake and what you learned
  • A time you hired someone great (or made a hiring mistake)
  • A time you had to prioritize ruthlessly — what got cut and why
  • A time you mentored or grew an engineer significantly
📡
Executive Communication
BLUF (Bottom Line Up Front). Speak in outcomes, not activities. Know how to present engineering health to non-technical leaders. Status = Green/Yellow/Red with context.
📊
Engineering KPIs
DORA metrics. Cycle time. Bug escape rate. Test coverage trends. On-call burden. Tech debt ratio. Developer happiness (eNPS). Know which matter most for your stage.
⚖️
Tradeoff Communication
Always present options, not just recommendations. "We can do X (fast, risky) or Y (slower, safer). I recommend X because..." Show you think in tradeoffs, not absolutes.
🔭
Technical Vision
Can you articulate a 1–3 year technical direction? What's your team's "north star" architecture? How do you align it with business goals? How do you communicate it to ICs?
⚠️
Risk Management
Identify, assess, and mitigate technical risk. RAID log (Risks, Assumptions, Issues, Dependencies). Know when to escalate vs absorb risk. Risk is the job, not the problem.
🎯
Prioritization
RICE, ICE, MoSCoW. Opportunity cost thinking. How to say no well. The difference between urgent and important. Backlog health as a leadership responsibility.
🏗️
Platform Thinking
When to build a platform vs a product. Internal developer experience. API-first. Golden paths. Paved roads. Reusable components vs bespoke solutions.
🔑
Ownership Mindset
Leaders own outcomes, not tasks. You own what happens on your watch — including what you didn't do. The difference between accountability and responsibility in a team context.
🔬
Operational Excellence
SLOs/SLAs/SLIs. Error budgets. Toil reduction. Automation as a leadership goal. Sustainable on-call. Capacity planning. Runbook culture.
